# 系统架构说明（可用于论文）

## 1. 后端服务层架构与构成

本系统的后端服务层主要基于 Python (Flask) 框架构建，负责整个数据工作流的调度、结构文件的解析、数据库操作以及外部计算插件的集成。后端服务层主要由以下三个核心模块构成：

### (1) 数据库与数据源
系统采用混合数据存储策略，结合了关系型数据库与结构化数据文件：
- **核心关系型数据库 (`pdbzn_workflow.db`)**：采用 SQLite 构建，用于记录整个 PDB-ZN 5步工作流（Step1-Step5）的执行状态、聚类结果、几何过滤参数以及蛋白主表（Master Table）的结构化核心字段。
- **静态与索引数据文件 (`backend/data/`)**：包含预先处理的 CSV 和 JSON 文件（如 `master_table.csv`、`pocket.csv`、`diffdock_index.json`），用于快速加载和检索前端所需的蛋白描述、对接得分和结合口袋参数。
- **结构文件库 (`PDB_ZN/`)**：本地存储的海量 3D 结构文件（`.pdb` / `.cif` 格式的受体文件，以及 `.sdf` 格式的小分子配体对接结果），由后端服务统一提供静态文件访问接口供前端三维渲染。

### (2) 外部计算插件 (Plugins)
为了实现复杂的生物信息学和计算化学分析，后端无缝集成了三大外部计算引擎：
- **TM-align**：用于蛋白质三维结构的局部与全局比对。后端在 Step2 聚类阶段调用该插件，计算蛋白代表序列与其成员之间的结构相似度（TM-score），从而剔除高度相似的冗余构象。
- **DiffDock**：基于扩散模型的分子对接插件。用于预测小分子配体与锌指蛋白/受体结合的最优构象，后端负责将大批量推理任务分配并解析其生成的 SDF 坐标结果。
- **fpocket**：用于蛋白质表面结合口袋（Binding Pockets）的几何形态检测与拓扑分析插件。通过分析残基的几何排布，输出预测口袋的体积、极性及疏水性等关键药物设计参数。

### (3) 核心代码块及其作用 (`diffdock_api_server.py`)
整个后端的业务逻辑集中在 `diffdock_api_server.py`，其主要包含以下功能代码块：
- **Web 服务路由层 (Routing Block)**：定义了所有的 RESTful API 接口（如 `/api/pdbzn/...`），接收来自前端的配置参数（如距离阈值、聚类模式），并将处理结果以 JSON 格式返回。
- **工作流引擎 (Workflow Engine)**：实现了从 Step1（预过滤）、Step2（TM-align 聚类）、Step3（配体/金属距离二次过滤）、Step4（3HIS 空间几何复核）到 Step5（主表合并）的顺序流水线逻辑。
- **几何拓扑计算模块 (Geometry Calculation Block)**：负责直接解析 `.cif`/`.pdb` 三维坐标数据，在 Step4 阶段精准测量组氨酸（HIS）与锌离子（ZN）之间的键长、键角，并筛选出 5Å 半径内的所有关键残基。
- **子进程任务调度器 (Subprocess Manager)**：异步管理和调度 TM-align、fpocket 等外部插件的命令行执行过程，捕捉运行日志并提取结构化输出，确保大运算量任务不会阻塞主服务器线程。

## 2. 前端页面架构及前后端交互机制

### (1) 前端页面架构
前端系统采用轻量级、多页面的单页/多页混合应用架构（HTML + Vanilla JavaScript + CSS），包含以下 **6 个核心页面**：
- **`index.html`（主页面）**：提供系统的全局入口。加载排名和 Master 数据，并通过内嵌的 `3Dmol.js` 插件实现蛋白质-配体复合物的三维可视化交互。
- **`pdbzn_workflow.html`（工作流控制台）**：PDB-ZN 数据库构建的核心页面，提供 5 个阶段（Step1-Step5）的参数配置表单、执行按钮与实时统计图表展示。
- **`master_table.html`（主表视图页）**：数据展示面板，提供针对最终筛选出的（如 70+ 个）目标蛋白的详细属性表格，支持多字段联合检索与过滤。
- **`diffdock_compare.html`（对接对比页）**：专门用于加载 DiffDock 推理结果，比对不同 Confidence 得分的分子对接构象。
- **`fpocket.html`（口袋分析页）**：展示通过 fpocket 计算出的药物结合口袋参数，并同步将预测的口袋在 3D 模型中进行高亮着色。
- **`tmalign.html`（结构比对页）**：可视化展示目标蛋白与其他参照序列之间的结构叠合情况与 TM-score。

### (2) 前后端交互机制
前后端之间采用 **RESTful API + 异步轮询 + 静态流传输** 的机制进行高效通信：
- **RESTful JSON 交互**：前端页面通过原生的 `fetch()` 或 `XMLHttpRequest` 发起 HTTP GET/POST 请求。例如，在执行 Step4 时，前端向后端发送包含 `require_his3: true` 参数的 JSON，后端处理后返回筛选成功的蛋白列表及丢弃统计信息的 JSON 响应。前端根据响应动态渲染 DOM（如更新页面上的饼图或统计数字）。
- **异步任务轮询 (Asynchronous Polling)**：对于执行 fpocket 或 TM-align 等耗时较长的操作，前端提交任务后会获得一个 `job_id`，随后前端使用定时器（`setInterval`）每隔几秒向后端请求该任务的当前进度。完成后端才会返回完整的处理结果，避免了 HTTP 超时。
- **三维数据流直传**：在所有涉及结构可视化的页面（如 `index.html`），前端 `3Dmol.js` 组件需要读取 `.pdb` 或 `.sdf` 数据。后端通过特殊的静态文件路由（如 `/api/files?path=...`），直接将服务器本地的大型文本结构文件以二进制流/纯文本形式返回给前端进行本地显卡加速渲染，极大地减轻了后端的内存负担。
